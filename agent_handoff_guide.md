# TPRM Hub вЂ” Agent Handoff Guide

## Project Summary

TPRM Hub is a modern, cloud-ready Third-Party Risk Management (TPRM) SaaS application that manages vendors, assets, assessments, and risks across an organization's supply chain. It is built as a full-stack application using Next.js 14 (App Router, TypeScript, Tailwind CSS, shadcn/ui) on the frontend and FastAPI (Python 3.12) with SQLAlchemy ORM, Alembic migrations, and a PostgreSQL database on the backend, all orchestrated via Docker Compose. The application is designed to run as a standalone tool and will later serve as a middleware layer between ServiceNow (ingesting New Application Requests) and OneTrust (syncing enriched vendor and assessment data). Phase 5 adds AI-powered security intelligence via the Anthropic Claude API, combining NVD/NIST CVE data, CISA KEV, EPSS, OSV, and HaveIBeenPwned scans with a Claude AI risk analysis layer. Refer to `tprm_plan.md` for the full project plan, data model, UI design system, and all build phases.

## Tech Stack & Conventions

### Frontend
- **Framework:** Next.js 14 App Router (`frontend/app/`)
- **Language:** TypeScript вЂ” all pages and components are fully typed
- **Styling:** Tailwind CSS with custom design tokens defined in `tailwind.config.ts` and `globals.css`
- **Components:** shadcn/ui primitives (`components/ui/`) plus feature components (`components/`)
- **All pages are client components** вЂ” every page file starts with `"use client"`
- **API calls:** All made via named functions in `frontend/lib/api.ts` using the generic `fetchApi<T>(path, options?)` wrapper; never call `fetch()` directly in a page
- **API routing:** Next.js rewrites all `/api/*` requests to the backend via `next.config.mjs` вЂ” frontend code always uses paths like `/api/v1/vendors`, never the raw backend URL
- **Query params:** Built with `URLSearchParams` inside typed `*Filters` interfaces (e.g. `VendorFilters`, `AssetFilters`)
- **File uploads:** Pass empty `headers: {}` to `fetchApi` for `FormData` bodies (do not set `Content-Type` manually for multipart)
- **State management:** React hooks only (`useState`, `useEffect`, `useCallback`, `useRef`) вЂ” no external state library
- **Type definitions:** All TypeScript interfaces in `frontend/lib/types.ts`
- **Routing conventions:** Dynamic params as `[id]`, query params via `useSearchParams()`, navigation via `next/link` and `useRouter().push()`

### Backend
- **Framework:** FastAPI with Python 3.12
- **ORM:** SQLAlchemy 2.0 (declarative models in `backend/app/models/`)
- **Migrations:** Alembic (`backend/alembic/versions/`) вЂ” always create a new migration file for schema changes
- **Schemas:** Pydantic v2 schemas in `backend/app/schemas/`; all schemas use `model_config = {"from_attributes": True}`
- **Services:** Business logic in `backend/app/services/` (currently empty, planned for Phase 3+)
- **AI integration:** `backend/app/ai/` (planned for Phase 5)
- **Router pattern:** Each module is an `APIRouter` with `prefix` and `tags`; routers are registered in `backend/app/main.py` under `/api/v1`
- **DB session:** Always injected via `db: Session = Depends(get_db)` вЂ” never instantiate directly
- **Enum handling:** Use `_to_values(payload.model_dump())` helper before passing to ORM constructors to convert Pydantic enum instances to their string values
- **ORM в†’ Pydantic:** Use `SchemaClass.model_validate(orm_obj)` for serialization
- **Error responses:** `raise HTTPException(status_code=..., detail="...")`
- **List endpoints:** Always return `PaginatedResponse[T]` with `items`, `total`, `limit` (default 25, max 100), `offset` (default 0)
- **Create endpoints:** Use `status_code=201`
- **Delete endpoints:** Use `status_code=204`
- **Search:** `ilike(f"%{search}%")` across relevant text fields using `or_(...)`
- **Ordering:** Default order by `name` for vendors/assets, `created_at.desc()` for time-ordered entities

### Database
- PostgreSQL 16 via Docker (service `postgres`, port 5432)
- All enum types defined as PostgreSQL native `ENUM` types in Alembic migrations
- Foreign keys use cascade deletes where appropriate
- All tables have `created_at` / `updated_at` with `server_default=sa.func.now()`
- Indexes on `name`, `vendor_id`, `entity_type`/`entity_id`, and `timestamp` columns

### API
- Versioned under `/api/v1/` from day one вЂ” never add unversioned routes
- All list endpoints paginated with `limit`/`offset`
- OpenAPI docs auto-generated by FastAPI at `http://localhost:8000/docs`
- Auth: JWT middleware via `HTTPBearer` scheme (`backend/app/core/auth.py`); `get_current_user()` dependency available but not yet enforced on all routes

### Design & Planning Decisions
- **Vendor category enum uses spaces, not underscores** вЂ” values are `"Cloud Provider"`, `"Software Vendor"`, `"Consultant"`, `"Hardware"`, `"Other"`. This is intentional and consistent across the DB enum, Pydantic schema, TypeScript types, and CSV import logic. Do not change to underscores.
- **NVD API key** вЂ” Add `NVD_API_KEY` to `.env` and `.env.example` before implementing Phase 5 NVD scanning. Request a free key at nvd.nist.gov/developers/request-an-api-key.
- **Anthropic API key** вЂ” `ANTHROPIC_API_KEY` is already in `.env` and `.env.example`. It is required for Phase 5 Claude AI Check features. The backend config reads it as an optional setting; it becomes required when the AI service is activated.
- **npm audit fix** вЂ” Only run `npm audit fix` (without `--force`). Never run `npm audit fix --force` вЂ” it can introduce breaking changes to locked dependency versions.
- **API proxy** вЂ” The frontend never calls the backend directly by host/port. All API calls go to `/api/v1/...` and Next.js rewrites them to `BACKEND_URL/api/v1/...` via `next.config.mjs`. This is required for Docker networking and must not be bypassed.
- **Assessment public token** вЂ” Generated by `secrets.token_urlsafe(32)[:64]` (URL-safe base64, stored in `public_token VARCHAR(64)`). Added via Alembic migration `0002_assessment_public_token.py`. A token is only generated on the first `POST /{id}/send` call and is reused on subsequent sends (Resend).
- **Assessment status flow** вЂ” `Draft` в†’ `Sent` (on send) в†’ `Completed` (on vendor submit). `In Progress`, `Overdue`, and `Closed` must be set manually via `PUT /{id}`. There is no automatic transition to "In Progress" on first token access.
- **Email** вЂ” Python's built-in `smtplib` (no extra pip dependency). Controlled by `email_enabled` in `.env`. When disabled (default), the public link is printed to the backend container's stdout. Vendor email is taken from `vendor.primary_contact_email`; if unset, the send still succeeds but no email is attempted.
- **PDF export** вЂ” Implemented as `GET /api/v1/assessments/{id}/export` returning an `HTMLResponse` with print-optimized CSS (`@media print`). Opened in a new browser tab from the detail page; user prints to PDF from the browser.
- **Public page layout** вЂ” `/public/*` routes render without the sidebar (Sidebar returns `null` when `pathname.startsWith("/public")`). The root layout's `<main>` padding still applies. No separate layout file is needed.
- **`ai_check` assessment type** вЂ” Creates an assessment with an empty questions list (`get_template_questions` returns `[]` for `ai_check`). The detail page shows "No questions for this assessment type." AI analysis logic is deferred to Phase 5.
- **Security scan background task** вЂ” `POST /security-scan` kicks off a background task via `run_in_threadpool(run_full_scan, asset_id)`, which runs scans sequentially. Each source updates its own `SecurityScanResult` record atomically. The frontend polls `GET /security-scan/status` every 2s until all sources are `completed` or `failed`.
- **NVD rate limiting** вЂ” Without `NVD_API_KEY`, NVD imposes strict rate limits (5 requests/30s). Add a free API key from nvd.nist.gov to `.env` to use the higher limit (50/30s). The `nvdlib` library handles back-off automatically.
- **Re-scan clears vulnerabilities** вЂ” `run_full_scan()` deletes all `asset_vulnerabilities` for the asset before running the NVD scan. CISA KEV and EPSS then enrich the freshly created records. This means a re-scan always reflects the latest state.
- **AI auto-created risks** вЂ” Auto-creation has been removed from both `nvd.py` (no longer creates Risk entries for Critical/High CVEs) and `ai_check.py` (no longer creates entries for `suggested_risks`). All CVE findings stay in `asset_vulnerabilities`. Users promote findings manually via the Security Intelligence Panel.
- **Security Intelligence Panel build order** вЂ” NVD в†’ CISA KEV в†’ EPSS в†’ OSV в†’ HaveIBeenPwned в†’ Claude AI Check (last, as it consumes results from all other sources).
- **Dashboard charts** вЂ” Implemented with pure SVG (donut charts) and CSS flexbox bars. No charting library (recharts, chart.js, etc.) is installed. The SVG donut uses `strokeDasharray`/`strokeDashoffset` on circles rotated -90В°. Keep this approach for Phase 7+ additions.
- **Vendor report** вЂ” `GET /api/v1/vendors/{id}/report` returns an `HTMLResponse` (same pattern as assessment export). The report opens in a new `<a target="_blank">` tag on the vendor profile page. `window.print()` fires automatically on load so the user can save as PDF immediately.
- **Notifications API** вЂ” `GET /api/v1/notifications/alerts` is a read-only endpoint (no side effects). `POST /notifications/send-overdue` triggers emails. Neither endpoint is wired to a scheduled job yet вЂ” callers must invoke them manually or via cron.
- **Audit log writes** вЂ” The `audit_logs` table exists and is read via `GET /api/v1/audit-logs`, but nothing currently writes to it. Phase 7 can add write calls in CRUD mutation endpoints using the pattern: `db.add(AuditLog(entity_type="vendor", entity_id=vendor.id, action="Create", changed_by=None, changes=payload.model_dump()))`.
- **Notifications SMTP settings** вЂ” `notifications.py` reuses the same email settings from Phase 4: `settings.email_enabled`, `settings.email_host`, `settings.email_port`, `settings.email_from`. These are already in `backend/app/core/config.py` and `.env`. No new environment variables needed for notification emails.
- **Assessment template questions sort_order** вЂ” The `assessment_template_questions` table uses `sort_order` (not `order`) as the column name to avoid SQL reserved word conflicts. Python attribute and DB column name are both `sort_order`.
- **Base template question count** вЂ” Low: 5q, Medium: 10q, High: 15q, Critical: 15q. Critical mirrors High because the existing `assessment_templates.py` service returned `_FULL` (15q) for both High and Critical. The templates were seeded from this service with no invented questions. Custom templates can add more questions on top.
- **Template soft delete** вЂ” `DELETE /api/v1/assessment-templates/{id}` sets `is_active=false` (soft delete). Base templates return 409 and cannot be deleted. Soft-deleted templates remain in the DB and can be restored by setting `is_active=true` via PUT.
- **Template question type mapping** вЂ” Old `assessment_templates.py` service used `yesno|text|multiline`. The new `templatequestiontype` enum uses `yes_no|text|multiple_choice|file_upload|rating`. Mapping on seed: `yesnoв†’yes_no`, `textв†’text`, `multilineв†’text` (no separate multiline type in the new system; the frontend QuestionBuilder uses a standard text input for both).
- **Risk Register vs. Security Intelligence separation** вЂ” CVE findings from the Security Intelligence Panel are never auto-created as Risk Register entries. They live in `asset_vulnerabilities` and are shown only in the Security Intelligence Panel. Users promote findings manually via a per-finding 'Promote to Risk Register' button or a bulk 'Add selected to Risk Register' action with per-finding checkboxes and a select-all option. This is intentional and must not be reverted.
